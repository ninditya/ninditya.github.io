01 — Context

A cinema dashboard that actually answers business questions.

CineTrack is a capstone project built during the Korea-ASEAN Digital Academy (KADA) program. My role was the backend API — specifically the analytics layer that powers an executive dashboard for a fictional cinema chain.

The frontend needed to show revenue breakdowns, occupancy rates, time-slot performance, and operational risk — all filterable by city, cinema, studio, movie, and date range. The backend had to handle all combinations efficiently.

02 — SQL Analytics Layer

Every business question is a SQL query.

Each analytics endpoint maps directly to a PostgreSQL query. The queries are not trivial SELECT * statements — they use CTEs, conditional aggregation, and date functions to compute business metrics on the fly.

Revenue by city — nested CTEs for correct occupancy

A naive query joining all tables and grouping by city gives wrong occupancy rates — because it averages schedule-level occupancy at the wrong granularity. The correct approach uses a two-layer CTE: first compute per-schedule stats, then aggregate to city level.

WITH filtered_schedules AS (
  SELECT s.schedule_id, st.total_capacity, c.city, c.cinema_id
  FROM schedules s
  JOIN studio st ON s.studio_id = st.studio_id
  JOIN cinema c  ON st.cinema_id = c.cinema_id
  -- WHERE clause injected dynamically from filter params
),
per_schedule AS (
  SELECT fs.city, fs.schedule_id, fs.total_capacity,
    COUNT(t.tiket_id)::int AS total_tickets,
    CASE WHEN fs.total_capacity > 0
      THEN COUNT(t.tiket_id)::float8 / fs.total_capacity::float8
      ELSE 0 END AS schedule_occupancy
  FROM filtered_schedules fs
  LEFT JOIN tiket t ON fs.schedule_id = t.schedule_id
  GROUP BY fs.city, fs.schedule_id, fs.total_capacity
)
SELECT city, COUNT(DISTINCT cinema_id)::int AS total_cinemas,
  SUM(total_tickets)::int, SUM(total_revenue),
  AVG(schedule_occupancy)::float8 AS avg_occupancy
FROM per_schedule
GROUP BY city ORDER BY SUM(total_revenue) DESC

Time slot analysis — EXTRACT and scoring

The time-slot endpoint computes demand and revenue per hour slot, then calculates a normalized optimization score for each slot. This powers pricing recommendations: high-demand, undermonetized slots get flagged for price increases.

LPAD(EXTRACT(HOUR FROM CAST(s.start_time AS TIME))::int::text, 2, '0') || ':00' AS time_slot

Weekend vs weekday — conditional day classification

CASE WHEN EXTRACT(DOW FROM CAST(s.show_date AS DATE)) IN (0, 6)
  THEN 'weekend' ELSE 'weekday'
END AS day_type

Dynamic filters — parameterized WHERE injection

Every endpoint accepts city, cinema_id, studio_id, movie_id, and payment_type filters. Rather than building unsafe string-concatenated queries, the backend uses a buildScopeFilters() utility that collects conditions and parameters separately, then injects the WHERE clause and passes params to query(sql, params).

03 — Caching & Performance

The same filter combination shouldn't hit the database twice.

Analytics queries join 4–5 tables with aggregations. Running them on every frontend page load is expensive. Redis caching wraps every endpoint with a filter-aware cache key.

The cache key is built from the endpoint name plus a JSON serialization of all filter parameters. The TTL is configurable via config.cacheTtlSeconds. When the cache hits, the database is bypassed entirely.

// Cache key: endpoint name + all filter params serialized
return withCache(
  "dashboard-sales-revenue-by-cinema",
  { start_date, end_date, city, cinema_id, top_n: String(top_n ?? "") },
  config.cacheTtlSeconds,
  async () => { /* actual DB query */ }
);

Query optimization with EXPLAIN ANALYZE

The dashboard's P1 bottleneck was the revenue-by-city query. After running EXPLAIN ANALYZE, the plan showed sequential scans on tiket.schedule_id and schedules.show_date. Adding composite indexes reduced the query from full-table scans to index range scans.

04 — LLM Integration

Raw data surfaced as business language.

Beyond the analytics queries, I integrated an LLM component that interprets the operational data and surfaces insights in plain language — making the dashboard useful to non-technical cinema managers, not just analysts.

  • Insight generation — The LLM receives aggregated metrics (top-performing slots, revenue gaps, occupancy trends) and returns human-readable recommendations for scheduling and pricing decisions.
  • Structured prompt design — The prompt includes tabular context (slot-level stats) alongside a structured output format, reducing hallucination risk on numeric claims.
  • Context injection — Business context (current filter scope, date range, comparison period) is injected into the prompt so recommendations are filter-aware, not generic.
05 — What building a backend taught me about data

SQL decisions upstream change what analysis is possible downstream.

Every analytics query I wrote revealed a schema decision that made it easier or harder. The two-layer CTE for occupancy wasn't clever engineering — it was the only way to get a correct answer given the schema. The occupancy metric is fundamentally a per-schedule ratio, not a per-city ratio.

Granularity matters

Aggregating occupancy at the city level by averaging ticket counts gives wrong results. You have to average at the right grain — schedule-level — then roll up. This is a common mistake in analytical SQL.

Dynamic filters are SQL injection risk

String concatenation for WHERE clauses opens injection. The solution is always parameterized queries — collect conditions and params separately, join conditions with AND, pass params to the driver.

EXPLAIN before you optimize

I guessed wrong about where the bottleneck was. EXPLAIN ANALYZE on the actual slow query showed the sequential scan on tiket — not where I expected. Measure first, optimize second.

AI Engineers need backend exposure

Understanding how an API shapes data — what's cached, what's computed on request, what's pre-aggregated — directly informs how I build ML pipelines and evaluation systems. The data doesn't exist in isolation.
