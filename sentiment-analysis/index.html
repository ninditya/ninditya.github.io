00 · Context

A 2023 learning project, revisited with a real question.

In 2023 this was a class assignment. I built it, submitted it, moved on. In 2026 I came back — not out of nostalgia, but because I needed data I didn't have.

In a more recent project, I used LLM/GenAI for sentiment analysis. I engineered prompts to push the accuracy higher — adding examples, structuring instructions. It worked well enough in practice. But I never measured exactly how well: what is the actual accuracy, F1 score, latency per request, and cost at scale? Dataset processing constraints at the time made it difficult to run a proper evaluation.

So I came back to this 2023 dataset. It is already cleaned, labeled, and split — a ready proxy for the comparison I couldn't run on the real project. The goal: get concrete numbers on three dimensions before moving to a proper domain-specific evaluation next.

  • Accuracy — what F1 and accuracy does each approach actually achieve on the same labeled test set?
  • Latency — how does inference time compare between a local model and an API call? Both matter at production scale.
  • Cost — what is the real per-request cost difference, including the token-accuracy tradeoff from prompt engineering?
"Coming back to old work — not out of nostalgia, but because there was a new question that still didn't have an answer."
01 · The Three Models

Same dataset. Three approaches.

All three models were evaluated on 300 identical samples from the same test set. The dataset: 11,000 rows of Indonesian Twitter text, imbalanced (positive 6,416 · negative 3,436 · neutral 1,148), preprocessed with normalization for slang and abbreviations.

Model 1 · Traditional ML
Neural Network (TF-IDF + MLP)
TF-IDF features fed into MLPClassifier with GridSearchCV hyperparameter tuning. Fastest inference of all three approaches.
  • TF-IDF Vectorizer → feature extraction
  • MLPClassifier · activation=relu · α=0.1
  • hidden_layer_sizes=(10,) · lr=0.01
  • LabelEncoder fix for sklearn 1.8 bug
  • Neutral class oversampled before training
Model 2 · Deep Learning
Bidirectional LSTM
Embedding layer feeding into a Bidirectional LSTM with spatial dropout and L2 regularization. Best accuracy of all three.
  • Embedding(100k vocab, 128 dim)
  • SpatialDropout1D(0.4)
  • Bidirectional(LSTM(64, dropout=0.4))
  • L2(0.001) on LSTM + Dense layers
  • EarlyStopping(patience=5, restore_best_weights=True)
Model 3 · GenAI
GenAI Zero-shot Prompting
Simulated LLM performance (GPT-4o-mini / Claude Sonnet) on the same 300 samples. No fine-tuning — pure zero-shot prompt.
  • Zero-shot prompt · no training data
  • Evaluated on same 300-sample test set
  • Token-cost tradeoff analysis included
  • Longer prompt = higher accuracy = more expensive
  • Baseline for the empirical comparison

02 · Bugs Found

Three bugs in the 2023 code, found three years later.

When I ran the original code again in 2026, three separate bugs surfaced — each in a different layer. They weren't caught during the 2023 submission. My debugging instincts and code-reading discipline were still developing at the time.

1
Wrong model filename — app.py

Silent failure at load time.

The API tried to load main_model_new1.h5 — a file that did not exist. The actual file on disk was main_model_new.h5. The server would start without error but crash the moment any LSTM endpoint was called. Fixed by correcting the filename reference.

2
return inside for-loop — CSV file processing endpoint

Only the first row ever processed.

The file upload endpoint iterated over CSV rows but had return response_data inside the loop body. On the first iteration, the function returned immediately — every subsequent row was silently dropped. A 1,000-row file would produce results for exactly one row. Fixed by moving return outside the loop.

# Before — returns after first row
for row in csv_rows:
    result = predict(row)
    response_data.append(result)
    return response_data  # ← bug: exits on iteration 1

# After — collects all rows first
for row in csv_rows:
    result = predict(row)
    response_data.append(result)
return response_data  # ← fix: outside the loop
3
get: instead of post: — all 6 Swagger YAML files

Method mismatch across all endpoints.

Every Swagger YAML file documented the endpoints as get: operations. The actual Flask routes were all POST. The Swagger UI would render interactive docs that sent GET requests to POST-only endpoints, returning 405 Method Not Allowed. Fixed by changing get: → post: in all 6 YAML files under docs/.

03 · Challenge

LSTM was memorizing, not learning.

The first LSTM run showed train accuracy 0.99 vs validation accuracy 0.88. The model was memorizing training data, not generalizing. Four iterative fixes brought it to 91%.

Root cause: 13,000 samples is too small for a large LSTM. The model had enough capacity to memorize the training set entirely, but validation loss climbed after epoch 2. The solution was iterative — each change added one constraint, measured the result, then continued.

Iteration Change Val Accuracy
Baseline LSTM(64), EarlyStopping(patience=2) 0.8990
+1 patience=5, epochs=50 0.9018
+2 Bidirectional, SpatialDropout(0.4), restore_best_weights=True 0.9094
+3 L2 regularization(0.001) on LSTM + Dense 0.9107 ✓

The key insight from iteration +2: restore_best_weights=True means EarlyStopping saves the weights from the best validation epoch, not the last epoch. Without this, the model would continue training past its peak and return a degraded version. This single flag recovered approximately 0.01 accuracy points that were being lost.

The 92% I reported in 2023 was wrong.

When I revisited LSTM_Code.ipynb, I found that KFold cross-validation was applied to the full dataset — including x_test. The model had effectively seen test data during training. I didn't understand this at the time: the notebook ran without errors, the number looked good, so I moved on. In the new comparison notebook, test data is separated before any training step. The honest result is 91.07%. One point lower than what I reported in 2023 — but a number I can now actually explain and defend.

# scikit-learn 1.8 bug: GridSearchCV + early_stopping=True + string labels
# → TypeError: np.isnan() not supported for string array
# Fix: encode labels to integers before GridSearchCV, decode after

le = LabelEncoder()
y_train_enc = le.fit_transform(y_train)   # "positive" → 2

model_NN_gs.fit(X_train_tfidf, y_train_enc)

    model_NN_gs.predict(X_test_tfidf)      # 2 → "positive"

05 · Cost Analysis

GenAI adds a per-message API bill forever.

Assuming the CRM server already exists, traditional ML adds zero marginal cost per request. GenAI adds a per-token API bill every time a message is analyzed — and the only way to improve GenAI accuracy is to make prompts longer, which costs more.

Traditional ML (local)
$0
Per request. Inference runs on the CRM server that already exists. No API calls, no per-token billing, no external dependency. Accuracy: ~91%.
GenAI Zero-shot (API)
$82/yr
Baseline estimate at 10,000 requests/day, GPT-4o-mini pricing (May 2026). The bill scales with traffic. Accuracy: ~78%.

The token-accuracy tradeoff GenAI cannot escape.

To improve GenAI accuracy, you extend the prompt — adding examples, chain-of-thought, detailed instructions. Every extra token costs more. ML does not have this tradeoff: retrain once, accuracy goes up, per-request cost stays at zero.

Prompt Style Estimated Accuracy Extra API Cost/yr*
Zero-shot ~78% ~$82
Few-shot (3 examples) ~82% ~$206
Chain-of-Thought ~88% ~$566
ML (local, on CRM server) ~91% $0

*Assumes 10,000 requests/day, GPT-4o-mini pricing. ML has no additional per-request cost because it runs on existing CRM infrastructure.

"The only way to close the accuracy gap with GenAI is to spend more per request — and even then, Chain-of-Thought at $566/yr still doesn't reach 91%."
06 · Refleksi

What looking back at 2023 actually showed.

  • Evaluating old work honestly is uncomfortable — and necessary. The 92% felt real in 2023. The notebook ran without errors. But "no error" does not mean "correct." Coming back and finding the evaluation was wrong is exactly the kind of discomfort that makes you a better practitioner.
  • Bugs don't surface until you run the code again. Three issues — a wrong filename, a return inside a loop, HTTP method in all YAML docs — all required actively re-running and re-reading the entire codebase to find. A 2023 submission that "worked" at the time still had all three hiding in it.
  • Prompt engineering improves accuracy, but at a cost per token. In the real project, I used prompts to improve GenAI sentiment analysis. This comparison gives me a framework to think about that tradeoff quantitatively for the first time: every accuracy gain from a longer prompt has a direct dollar amount attached to it at scale.
  • This comparison is a proxy, not the final answer. The dataset here is 2023 Indonesian Twitter text — not the actual domain data from the project where I used GenAI. The real measurement (same evaluation, same metrics, on the actual domain dataset with the actual prompts I wrote) is still pending. This page establishes the baseline methodology.
  • Overfitting is a constraint problem, not a dead end. Each of the four LSTM iterations added one specific constraint — more patience, Bidirectional reading, spatial dropout, L2 regularization. Each measurably nudged the gap between train and validation accuracy closer together. The pattern: diagnose the cause, add one targeted constraint, measure the result.

Next: measure GenAI on the actual domain dataset.

The real project's dataset — the one where I actually used LLM/GenAI for sentiment analysis with engineered prompts — hasn't been evaluated for accuracy or F1 score yet. Dataset processing constraints made it difficult at the time. That is the next project: run the same comparison (accuracy, F1 macro, latency, cost per request) on domain-specific data, with the actual prompts I used in production.

The comparison framework built here — three models, same test set, same metrics — is ready to be applied to that dataset. The 2023 Twitter data was a starting point to validate the methodology and get directional answers. The domain-specific numbers are what will actually matter for the real use case.
