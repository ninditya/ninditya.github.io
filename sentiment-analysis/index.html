00 · Why This Comparison

A real question from work, answered with real data.

At a work project, I was the one managing and adjusting the prompt for a GenAI-based sentiment analysis system. Results varied — sometimes the classification felt off. But I had no measurement framework to know how off, or whether my prompt adjustments were genuinely helping.

I had already built a sentiment analysis model in 2023 — Neural Network and LSTM on Indonesian Twitter data. In 2026, I extended it into a proper comparison: both approaches, same dataset, same metrics, same 300-sample test set.

  • ML pipeline — EDA → preprocessing → oversampling → modeling → overfitting fix / GridSearch → evaluation and insights
  • GenAI prompting — temperature, zero-shot vs few-shot vs CoT, real API inference, cost at scale, evaluation and insights
  • Head-to-head comparison — accuracy, F1, latency, and per-request cost on the same 300 labeled samples
"I was adjusting prompts at work without knowing if they were making things better or worse. This comparison finally gave me that answer."
01 · The Approaches

ML pipeline and GenAI prompting, built and measured.

The dataset: 11,000 rows of Indonesian Twitter text, labeled positive · negative · neutral. Heavily imbalanced — positive dominates at 6,416 rows, neutral is the minority class at 1,148. This imbalance shapes every decision in the ML pipeline.

EDA: what the data looks like before any modeling.

Three things from the data that shaped every decision downstream:

  • Class imbalance — positive 58%, negative 31%, neutral 10%. Without oversampling, neutral recall collapses.
  • Informal language — slang ("gue", "gapapa"), abbreviations ("tdk", "udh"), mixed case. Standard NLP pipelines break without a domain-specific normalization layer.
  • Short texts — tweet-length, 5–30 tokens after cleaning. Enough signal for TF-IDF; LSTM needs careful regularization to not memorize patterns.

Neural Network (TF-IDF + MLP)
TF-IDF features fed into MLPClassifier with GridSearchCV hyperparameter tuning. Fastest inference of all three approaches.

Oversampling: fixing the class imbalance before training

Neutral class: 1,148 rows → balanced.

Without correction, any model learns that predicting "positive" most of the time gets decent accuracy — without learning neutral or negative properly. The fix: duplicate neutral samples until the training set is balanced. One limitation: oversampling happened before the train-test split, so duplicate samples could appear in both sets.

The GenAI approach: prompting, temperature, real API inference.

Each text is sent to an LLM with a written prompt; the model returns a label. No training — just OpenRouter API with Gemini 2.0 Flash Lite, on the same 300-sample test set.

Temperature

At temperature=0, the model always picks the most probable token — consistent, reproducible output. Higher values introduce randomness. For a 3-label classification task, low temperature is the right choice. The effect across different temperature values is in the chart at section 05.

Prompting strategy

Three styles tested: zero-shot, few-shot (3–10 examples), chain-of-thought. The surprising result: adding examples made accuracy worse. The few-shot examples were e-commerce reviews — a different domain from Twitter. The model anchored to the wrong patterns. Zero-shot won on accuracy and cost.

02 · Bugs Found

Three bugs in the 2023 code, found three years later.

When I ran the original code again in 2026, three separate bugs surfaced — each in a different layer. They weren't caught during the 2023 submission. My debugging instincts and code-reading discipline were still developing at the time.

1
Wrong model filename — app.py

Silent failure at load time.

The API tried to load main_model_new1.h5 — a file that did not exist. The actual file on disk was main_model_new.h5. The server would start without error but crash the moment any LSTM endpoint was called. Fixed by correcting the filename reference.

2
return inside for-loop — CSV file processing endpoint

Only the first row ever processed.

The file upload endpoint iterated over CSV rows but had return response_data inside the loop body. On the first iteration, the function returned immediately — every subsequent row was silently dropped. A 1,000-row file would produce results for exactly one row. Fixed by moving return outside the loop.

# Before — returns after first row
for row in csv_rows:
    result = predict(row)
    response_data.append(result)
    return response_data  # ← bug: exits on iteration 1

# After — collects all rows first
for row in csv_rows:
    result = predict(row)
    response_data.append(result)
return response_data  # ← fix: outside the loop
3
get: instead of post: — all 6 Swagger YAML files

Method mismatch across all endpoints.

Every Swagger YAML file documented the endpoints as get: operations. The actual Flask routes were all POST. The Swagger UI would render interactive docs that sent GET requests to POST-only endpoints, returning 405 Method Not Allowed. Fixed by changing get: → post: in all 6 YAML files under docs/.

03 · Challenge

LSTM was memorizing, not learning.

The first LSTM run showed train accuracy 0.99 vs validation accuracy 0.88. The model was memorizing training data, not generalizing. Four iterative fixes brought it to 91%.

Root cause: 13,000 samples is too small for a large LSTM. The model had enough capacity to memorize the training set entirely, but validation loss climbed after epoch 2. The solution was iterative — each change added one constraint, measured the result, then continued.

Iteration Change Val Accuracy
Baseline LSTM(64), EarlyStopping(patience=2) 0.8990
+1 patience=5, epochs=50 0.9018
+2 Bidirectional, SpatialDropout(0.4), restore_best_weights=True 0.9094
+3 L2 regularization(0.001) on LSTM + Dense 0.9089 ✓

The key insight from iteration +2: restore_best_weights=True means EarlyStopping saves the weights from the best validation epoch, not the last epoch. Without this, the model would continue training past its peak and return a degraded version. This single flag recovered approximately 0.01 accuracy points that were being lost.

Train accuracy terus naik mendekati 0.97 sementara validation stagnan di 0.90 — tanda klasik overfitting pada dataset kecil.

The 92% I reported in 2023 was wrong.

When I revisited LSTM_Code.ipynb, I found that KFold cross-validation was applied to the full dataset — including x_test. The model had effectively seen test data during training. I didn't understand this at the time: the notebook ran without errors, the number looked good, so I moved on. In the new comparison notebook, test data is separated before any training step. The honest result is 91.07%. One point lower than what I reported in 2023 — but a number I can now actually explain and defend.

# scikit-learn 1.8 bug: GridSearchCV + early_stopping=True + string labels

05 · Cost Analysis

GenAI adds a per-message API bill forever.

Assuming the CRM server already exists, traditional ML adds zero marginal cost per request. GenAI adds a per-token API bill every time a message is analyzed — and making the prompt longer doesn't even help.

Traditional ML (local)
$0
Per request. Inference runs on the CRM server that already exists. No API calls, no per-token billing. Accuracy: 93.0% (300-sample eval).
GenAI Zero-shot (API)
$62/yr
Baseline at 10,000 requests/day, Gemini 2.0 Flash Lite ($0.075/1M input token, May 2026). Bill scales with traffic. Accuracy: 85.3%.

Surprising finding: longer prompt = worse accuracy.

The expected pattern — more examples lead to better results — didn't hold here. Few-shot and chain-of-thought prompts actually hurt accuracy compared to zero-shot. Root cause: the few-shot examples used e-commerce reviews, while the test data is informal Twitter text. The mismatch confused the model. Zero-shot turned out to be both the most accurate and the cheapest GenAI strategy.

Why zero-shot > few-shot in this case

Few-shot only works when the examples match the domain.

The few-shot examples I hardcoded in the prompt were product reviews — things like "produk bagus kualitas oke, pengiriman cepat." Formal sentences, e-commerce vocabulary, complete grammar. The test data is entirely different: Indonesian Twitter with slang, abbreviations, emoticons, and incomplete sentences.

When you give a model examples from the wrong domain, it doesn't ignore them — it anchors to those patterns. When it encountered a tweet like "gapapa sih tp kurang puas", the e-commerce-style anchors actively got in the way. Zero-shot has no examples to anchor to, so it relies on general language understanding instead. Out-of-domain few-shot examples aren't extra information — they're noise that costs more money.

Zero-shot mendominasi sudut optimal — akurasi terbaik GenAI sekaligus biaya termurah. Few-shot dan CoT lebih mahal tapi akurasinya justru turun karena model ter-anchor ke pola e-commerce yang salah domain. ML (bintang biru) jauh di atas semua.

Pengaruh temperature GenAI terhadap akurasi. Temperature rendah (0.0) lebih deterministik dan stabil; temperature tinggi meningkatkan variance output — relevan saat memilih setting deployment.

"Spending more on GenAI prompts doesn't just add cost — it actively makes the model worse. Zero-shot wins on accuracy and price. ML wins on both by an even wider margin."
06 · Refleksi

What looking back at 2023 actually showed.

  • Evaluating old work honestly is uncomfortable — and necessary. The 92% felt real in 2023. The notebook ran without errors. But "no error" does not mean "correct." Coming back and finding the evaluation was wrong is exactly the kind of discomfort that makes you a better practitioner.
  • Bugs don't surface until you run the code again. Three issues — a wrong filename, a return inside a loop, HTTP method in all YAML docs — all required actively re-running and re-reading the entire codebase to find. A 2023 submission that "worked" at the time still had all three hiding in it.
  • Prompt engineering improves accuracy, but at a cost per token. In the real project, I used prompts to improve GenAI sentiment analysis. This comparison gives me a framework to think about that tradeoff quantitatively for the first time: every accuracy gain from a longer prompt has a direct dollar amount attached to it at scale.
  • This comparison is a proxy, not the final answer. The dataset here is 2023 Indonesian Twitter text — not the actual domain data from the project where I used GenAI. The real measurement (same evaluation, same metrics, on the actual domain dataset with the actual prompts I wrote) is still pending. This page establishes the baseline methodology.
  • Overfitting is a constraint problem, not a dead end. Each of the four LSTM iterations added one specific constraint — more patience, Bidirectional reading, spatial dropout, L2 regularization. Each measurably nudged the gap between train and validation accuracy closer together. The pattern: diagnose the cause, add one targeted constraint, measure the result.

Next: retry few-shot with in-domain examples.

The zero-shot win comes with an asterisk: the few-shot examples were out-of-domain (e-commerce, not Twitter). The next experiment is replacing them with labeled tweets from the same distribution — to test whether few-shot can actually beat zero-shot when the examples match. After that: run the same comparison on the actual work dataset, the one where these questions originally came from.
