01 — Problem

Attrition above 10% is a management warning sign. Theirs was 16.92%.

Jaya Jaya Maju operates across multiple countries, employing 1,000+ staff. Their attrition rate — 16.92%, or roughly 179 employees per year — was significantly above the acceptable threshold. No structured early-warning system existed.

Attrition is expensive. Replacing an employee costs an estimated 50–200% of their annual salary when recruitment, onboarding, and lost productivity are accounted for. At 179 employees per year, the financial exposure is substantial — and most of it is preventable with early identification.

The dataset contains 72 features covering four domains:

  • Work conditions — Overtime requirement, business travel frequency, distance from home. These are the strongest controllable risk factors.
  • Compensation — Monthly income, daily rate, hourly rate, salary hike percentage, stock option level. Underpayment relative to peers is a key retention lever.
  • Satisfaction scores — Job satisfaction, environment satisfaction, work-life balance score, job involvement. Self-reported scores that predict behavior.
  • Career trajectory — Total working years, years at company, years in current role, years since last promotion, training frequency. Stagnation is a warning sign.

02 — Why XGBoost Won

Recall beats accuracy when missing a leaver costs more than a false alarm

Two models were compared: Logistic Regression (interpretable baseline) and XGBoost (non-linear ensemble). The right choice depended on what kind of error was more expensive.

Metric Logistic Regression XGBoost (Selected)
Test Accuracy 74.53% 59.91%
Test Recall (Attrition) 72.22% 86.11% ✓
ROC-AUC 79.92% 81.93% ✓
False Negatives More Only 5

XGBoost has lower raw accuracy but dramatically higher recall. In an HR context, a false negative (missing a leaver) means losing an employee without any chance to intervene. A false positive (flagging someone who stays) costs a counseling conversation. The asymmetry clearly favors recall — XGBoost was chosen.

SMOTE was applied to training data only. Class imbalance (roughly 83% stay / 17% leave) would have biased any model toward always predicting "no attrition." SMOTE synthesizes minority-class examples during training to force the model to learn both distributions.

04 — Key Findings

The pattern: workload + underpayment + stagnation = attrition

Overall attrition
16.92%
179 out of 1,058 labeled employees left. This is the baseline the model aims to reduce through early identification.
Overtime risk
31.92%
The single strongest controllable factor. Employees required to work overtime are nearly 3× more likely to leave. Policy change here has the highest leverage.
Salary quartile gap
29% vs 9%
Bottom salary quartile loses 29% annually. Top quartile loses 9.81%. The retention premium of competitive pay is clearly visible in the data.
New employee risk
34.59%
First-year attrition is the highest of any tenure band. Onboarding quality and early job fit are more critical than later retention programs.
Work-life balance
32% vs 10%
Score 1 (poor) leads to 32.14% attrition. Score 4 (good) drops below 10%. Self-reported satisfaction scores are genuine predictors, not just HR survey noise.
Model false negatives
5
XGBoost missed only 5 actual attritors on the test set. In an HR intervention context, this means the model's flags are reliable enough to act on.
05 — Deliverables

From model to operational HR infrastructure

Metabase Dashboard (5 Tabs)

Built on Docker + PostgreSQL, accessible to HR teams through the browser:

  • Executive Summary — KPIs (attrition rate, department breakdown, role breakdown), filterable by department and role
  • Compensation & Workload — Overtime distribution, business travel impact, salary band attrition rates
  • Satisfaction & Engagement — Job satisfaction scores, environment satisfaction, work-life balance, job involvement correlations
  • Demographics & Career — Age distribution, marital status, tenure bands, promotion recency, training times
  • Early Warning System — Model predictions per employee with interactive filters for HR to prioritize outreach

Batch Prediction Script

Production-ready Python script for recurring HR use:

  • Input — Accepts CSV of employee records with the 72 feature columns
  • Processing — Custom HRFeatureEngineer transformer handles age grouping, binary encoding, ordinal encoding, one-hot encoding in a scikit-learn pipeline
  • Output — Returns attrition probability (0–100%) and binary prediction per employee
  • Database integration — SQLAlchemy connector to write results directly to PostgreSQL, populating the Metabase Early Warning tab

Infrastructure: Docker Compose orchestrates PostgreSQL + Metabase containers. The entire analytics environment spins up with a single docker compose up.

Beginner (4 courses) Belajar Matematika untuk Data Science → Belajar Penerapan Data Science — Submission 1 (this project) Submission 2 (Student Dropout)

Reviewer feedback — two rounds of rejection.

First rejected for missing infrastructure, second rated ★4 for generic recommendations:

Feedback #1 — Missing Infrastructure File

The command python setup_database.py failed because the file was not attached. Reviewer could not verify the dashboard at all. No matter how good the model or the Metabase setup is — if someone can't reproduce it, it doesn't exist from the evaluator's perspective.

Feedback #2 — README Completeness

The README needed: dataset source link (so the reviewer can verify the data), Python version (so the environment can be reproduced), and step-by-step virtual environment setup (create → activate → install from requirements.txt).

Why Not ★5 — Action Items Were Too Generic

Recommendations like "evaluate compensation" or "improve work-life balance" are valid in principle but not grounded in the specific data from this analysis. A ★5 action item looks like: "Employees in Sales Representative roles earning below [X] with overtime have a >70% attrition probability." Each recommendation needs a segment, a number, and an implementation step — not a category.

Future Development

Based on reviewer feedback and the direction this analysis points toward, here's what the next iteration of this project will include:

Data-Backed Recommendations

Add a segment × metric table to every action item — e.g., "Sales Representatives earning below Q1 with overtime have >70% predicted attrition risk." Each recommendation includes a specific employee segment, measured risk level, and estimated retention impact.

What-If Simulation

Add a Streamlit interface where HR can simulate: "if overtime is eliminated for this department, how does predicted attrition change?" Turns the passive dashboard into an active decision tool.

Explainability (SHAP)

Integrate SHAP values to show per-employee why the model flagged them — not just "high risk" but "high risk because: overtime=Yes, tenure=0.8yr, salary=low quartile." Makes predictions actionable at the individual level.

Employee ID in Output

The batch prediction output needs an employee ID column so HR can map predictions back to individuals in their HR system. Simple to add — critical for operationalization.
